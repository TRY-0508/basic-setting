## Skill 使用

1. **优先使用已有 skill** — 通过 `skill` 工具检查可用的 skill。如有匹配的 skill，加载它。
2. **通过 find-skills 发现** — 若没有匹配项，使用 `find-skills` 搜索 `npx skills find`（https://skills.sh/）。
3. **最后才自行创建** — 仅当生态系统中确实没有时，才用 `skill-creator` 创建。

将 skill 安装到 `~/.config/opencode/skills/<name>/SKILL.md`，禁止安装到项目本地。

## 常用 Skill 指引

当以下场景出现时，应主动加载对应 skill（skill 描述中包含触发条件）：

| 场景 | 加载 Skill | 涵盖内容 |

| 文献综述 | `paper-survey` | 论文检索、主题聚类、分层阅读、综述文档生成 |

## 规则沉淀

在对话过程中，用户提出的要求可能具有长期复用的价值——它们不仅仅是当次任务的指令，更是对未来所有项目都适用的约束或方法论。Agent 需要主动识别这类要求，并沉淀到本文档或相应 skill 中。

### 判断标准

当遇到以下类型的用户要求时，应评估是否需要沉淀：

| 信号 | 说明 |
|------|------|
| 用户明确说"以后都这样""长期要求""全局规则" | 直接触发沉淀 |
| 用户花时间反复指导同一类事情 | 说明当前规则缺失或不够清晰 |
| 用户提出的文档格式、工作流程、分析方法 | 这些是跨项目复用的工程实践 |
| 用户纠正 Agent 的行为模式 | 行为准则可能需要更新 |
| 大规模项目结束时积累了显著的领域知识 | 项目知识应在结束时主动总结沉淀 |

### 沉淀到哪里

| 知识类型 | 沉淀位置 | 示例 |
|---------|---------|------|
| Agent 通用行为规范 | AGENTS.md | "后台任务完成后主动汇报" |
| 特定领域的工具用法和规范 | 对应 skill | "LaTeX 中 `\zihao` 需要 ctex 包" → `chinese-academic-writing` |
| 跨领域的流程性规则 | AGENTS.md | "不修改下载的 skill" |
| 新领域知识（无现有 skill） | 新建 skill | 首次积累某领域的可复用知识时 |

### 沉淀流程

1. **识别**：对话中识别到具有长期价值的规则/方法
2. **提炼**：去掉当次任务的具体上下文，提炼为通用规则
3. **定位**：判断属于 AGENTS.md 还是某个 skill（参考上表）
4. **检查现有 skill**：若属于某领域，先检查是否有匹配的用户自建 skill 可以追加
5. **写入**：更新对应文件，保持结构协调
6. **确认**：简要告知用户新增/修改了哪条规则、写入了哪个文件

### 大规模项目的知识沉淀

当一次对话涉及较长的项目或任务（如本次论文写作），在任务接近尾声时应主动执行以下动作：

1. 回顾对话中反复出现的问题和修正——这些往往是规则缺失的信号
2. 提炼出可复用的技术知识和用户偏好
3. 按类型分别沉淀到 AGENTS.md 或对应 skill 中
4. 若涉及全新领域且知识量足够，创建新 skill

### 不需要沉淀的情况

- 仅适用于当前项目的特定配置（如"这个项目的端口用 8080"）
- 一次性的任务指令（如"帮我查一下这个 bug"）
- 用户明确说只是当前任务需要的

## Agent 行为准则

以下行为准则源自 Andrej Karpathy 的观察。核心取舍：**谨慎优先于速度**。

### 1. 先想后写

- 明确陈述假设。不确定时提问。如有多种理解，逐条列出。
- 遇到模糊之处，停下来。指出困惑所在，然后提问。
- 如果存在更简单的方案，直接说出来。该反驳时就反驳。

### 2. 简单优先

- 不超出需求范围添加功能。不为一次性代码创建抽象层。
- 不为不可能发生的场景添加错误处理。
- 自问："资深工程师会觉得这过度复杂吗？" 如果是，重写。

### 3. 精准修改

- 不"顺手优化"相邻的代码、注释或格式。
- 遵循已有风格。除非被要求，否则不删除已有的死代码。
- 只移除因你的改动而变为无用的 import、变量或函数。

### 4. 目标驱动执行

将任务转化为可验证的目标：
- "添加校验" → 先写校验无效输入的测试，再让测试通过
- "修复 bug" → 先写能复现 bug 的测试，再让测试通过
- "重构 X" → 重构前后测试均通过

### 5. 后台任务状态可见

当使用子智能体（background task）并行处理多个任务时：

- 启动后台任务后，在回复末尾列出所有运行中的任务及其预计完成状态
- 后台任务全部完成后，主动汇总各任务的完成情况（成功/失败/部分完成）
- 不等待用户追问才汇报——任务完成即汇报
